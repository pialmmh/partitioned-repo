Split-Verse Documentation

Quick Start → Architecture → Examples → GitHub →

Split-Verse is a distributed sharding framework for MySQL that provides horizontal scaling through database-level sharding and table-level partitioning. It enforces best practices for distributed systems while maintaining simplicity and performance.

🚀 Infinite Scalability

Scale horizontally by adding shards dynamically. Each shard can handle millions of records across thousands of partitioned tables.

🎯 Smart Routing

Two-level routing system: hash-based shard selection and value-based partition selection for optimal query performance.

🔧 Simple API

Builder-pattern configuration with type-safe compilation guarantees. No complex XML or annotations required.

⚡ High Performance

Direct table access for ID+partition queries. Parallel execution across shards. Efficient batch operations.

Key Features

  • Single Primary Key No composite keys in database schema
  • External IDs UUID, ULID, or NanoID generation (no AUTO_INCREMENT)
  • Flexible Partitioning Time-based, hash-based, or value-based partitioning
  • Multi-Table Support Separate tables instead of native MySQL partitions
  • Auto Maintenance Automatic partition creation and deletion

When to Use Split-Verse

Perfect For:

  • Time-series data (logs, events, metrics)
  • Multi-tenant SaaS applications
  • High-volume transactional systems
  • Applications requiring geographic distribution

Not Suitable For:

  • Applications requiring complex JOINs across shards
  • Systems needing strong consistency across all data
  • Small datasets that fit in a single server

Quick Start Guide

Installation

Maven

<dependency>
    <groupId>com.telcobright</groupId>
    <artifactId>split-verse</artifactId>
    <version>1.0.0</version>
</dependency>

Gradle

implementation 'com.telcobright:split-verse:1.0.0'

Basic Example

// 1. Define your entity
public class Order implements ShardingEntity {
    private String id;
    private LocalDateTime createdAt;
    private String customerId;
    private BigDecimal amount;

    // Implement required methods
    @Override
    public String getId() { return id; }
    @Override
    public void setId(String id) { this.id = id; }
    @Override
    public LocalDateTime getCreatedAt() { return createdAt; }
    @Override
    public void setCreatedAt(LocalDateTime createdAt) {
        this.createdAt = createdAt;
    }
}

// 2. Configure repository
ShardingRepository<Order> repository = SplitVerseRepository.<Order>builder()
    .withEntityClass(Order.class)
    .withTableName("orders")
    .withShardingStrategy(ShardingStrategy.DUAL_KEY_HASH_RANGE)
    .withPartitionColumn("created_at", PartitionColumnType.LOCAL_DATE_TIME)
    .withPartitionRange(PartitionRange.DAILY)
    .withRepositoryMode(RepositoryMode.MULTI_TABLE)
    .withDataSources(Arrays.asList(
        DataSourceConfig.create("shard1.db.com", 3306, "db1"),
        DataSourceConfig.create("shard2.db.com", 3306, "db2")
    ))
    .build();

// 3. Use the repository
Order order = new Order();
order.setId(NanoIdUtils.randomNanoId());
order.setCreatedAt(LocalDateTime.now());
order.setCustomerId("customer-123");
order.setAmount(new BigDecimal("99.99"));

repository.insert(order);

// 4. Query data
Order found = repository.findById(order.getId());
List<Order> todayOrders = repository.findAllByDateRange(
    LocalDateTime.now().withHour(0).withMinute(0),
    LocalDateTime.now()
);

Configuration Options

Option Required Description
withEntityClass Yes Entity class implementing ShardingEntity
withTableName Yes Base table name
withShardingStrategy Yes SINGLE_KEY_HASH, DUAL_KEY_HASH_RANGE, or DUAL_KEY_HASH_HASH
withPartitionColumn Conditional Required for DUAL_KEY strategies
withRepositoryMode No PARTITIONED or MULTI_TABLE (default)
withDataSources Yes List of database connections

Design Principles

1. Single Primary Key Only

  • Every table has a single VARCHAR(8-22) primary key
  • No composite keys in database schema
  • Simplifies ORM integration and migrations

2. External ID Generation

  • IDs must be generated externally (UUID, ULID, NanoID)
  • No AUTO_INCREMENT columns
  • Ensures consistent hashing across shards

3. Two-Level Distribution

  • Level 1 (Sharding): Hash-based distribution across database servers
  • Level 2 (Partitioning): Value-based or hash-based table selection

Query Resolution Flow

Query: findByIdAndDateRange("order-123", startDate, endDate)
    ↓
Step 1: Hash("order-123") → Shard 2
    ↓
Step 2: Date range → Tables [orders_2024_01_01, orders_2024_01_02]
    ↓
Step 3: Execute on Shard 2:
    SELECT * FROM orders_2024_01_01 WHERE id = ? AND created_at BETWEEN ? AND ?
    UNION ALL
    SELECT * FROM orders_2024_01_02 WHERE id = ? AND created_at BETWEEN ? AND ?
    ↓
Step 4: Merge results and return

Sharding Concepts

Sharding distributes data across multiple database servers (shards) to achieve horizontal scalability. Split-Verse uses consistent hashing on the entity ID to determine shard placement.

How Sharding Works

  1. ID Generation: Application generates a unique ID (UUID, ULID, NanoID)
  2. Hash Calculation: ID is hashed using MurmurHash3
  3. Shard Selection: Hash modulo shard count determines target shard
  4. Query Execution: Operations execute on selected shard

// Configure multiple shards
List<DataSourceConfig> shards = Arrays.asList(

Adding New Shards

Steps to Add a New Shard:

  1. Provision new database server
  2. Create database schema
  3. Update configuration with new shard
  4. Restart application (new data goes to all shards)
  5. Optional: Rebalance existing data

Partitioning Strategies

Partitioning divides tables within each shard into smaller, manageable pieces. Split-Verse supports multiple partitioning strategies based on your data access patterns.

Partition Types

1. Time-Based Partitioning

// Daily partitions for time-series data
.withPartitionColumn("created_at", PartitionColumnType.LOCAL_DATE_TIME)
.withPartitionRange(PartitionRange.DAILY)

// Results in tables like:
// orders_2024_01_01
// orders_2024_01_02
// orders_2024_01_03

2. Hash-Based Partitioning

// Hash partitioning for even distribution
.withPartitionColumn("customer_id", PartitionColumnType.STRING)
.withPartitionRange(PartitionRange.HASH_16)

// Results in tables like:
// orders_hash_00
// orders_hash_01
// ...
// orders_hash_15

3. Value-Based Partitioning

// New table every 1 million records
.withPartitionColumn("sequence_number", PartitionColumnType.LONG)
.withPartitionRange(PartitionRange.VALUE_RANGE_1M)

// Results in tables like:
// orders_range_0      // 0-999,999
// orders_range_1      // 1,000,000-1,999,999
// orders_range_2      // 2,000,000-2,999,999

Partition Range Options

Range Type Use Case Table Naming
HOURLY High-volume logs table_2024_01_01_00
DAILY Orders, events table_2024_01_01
MONTHLY Reports, aggregations table_2024_01
YEARLY Archives table_2024
HASH_N Even distribution table_hash_00
VALUE_RANGE Numeric ranges table_range_0
